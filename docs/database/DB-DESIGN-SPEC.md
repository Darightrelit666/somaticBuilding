# 数据库设计权威规范（项目统一口径）

来源：用户提供（可用于 NotebookLM 投喂），为本项目数据库设计强制规范。

---

## 一、通用顶层权威原则（所有库都遵守）
- 三大范式优先：1NF/2NF/3NF 打底，防冗余、防更新异常；高并发场景适度反范式（空间换性能）。
- ACID 强制落地：核心业务必须走事务，严格遵从原子性 / 一致性 / 隔离性 / 持久性。
- **InnoDB 唯一选型（MySQL）**：禁用 MyISAM，必用事务、行锁、MVCC、崩溃恢复。
- 冷热分离：热数据主库 + 缓存，冷数据归档 / 分库 / 大数据仓，不堆一张大表。

---

## 二、库 & 表命名规范（阿里 / 字节官方通用）
- 数据库名：小写 + 下划线，业务前缀拆分  
  例：`crm_db`、`order_db`、`user_db`
- 表名：模块_业务_场景，全小写、禁止拼音、禁止特殊字符  
  例：`order_main`、`user_account`、`goods_sku`
- 禁用：单字母表、生僻缩写、大小写混用、系统关键字（name/order/user）

---

## 三、字段设计权威规范（硬核必守）
### 1. 基础约束
- 核心字段 NOT NULL，禁止大量 NULL（索引、统计、MVCC 都坑）
- 必须有：主键、创建时间、更新时间、逻辑删除标记

### 2. 主键规范
- 优先 自增 BIGINT / 雪花算法 ID；禁用 UUID 当主键（索引碎片大、插入慢）
- 主键无业务含义，绝不拿手机号 / 身份证做主键

### 3. 数据类型严控（大厂踩坑版）
- 整型：能用 INT 不 BIGINT，能用 TINYINT 不 INT（省空间、提升缓存命中率）
- 字符串：手机号 CHAR(11)、固定编码用 CHAR、变长用 VARCHAR；超长文本用 TEXT（不滥用）
- 时间：必用 DATETIME(3) 存毫秒；禁用 TIMESTAMP（时区 / 范围坑）
- 金额：DECIMAL (18,2) 强制，禁止浮点 float/double（精度丢失致命）

### 4. 字段命名
- 下划线命名：`user_id`、`order_amount`、`create_time`
- 状态统一：`status`（0 禁用 / 1 正常），统一字典，全库口径一致

---

## 四、索引设计权威规范（MySQL 官方 + 阿里规约）
- 主键必建聚簇索引
- 高频查询字段建索引，单表索引不超 5 个（写性能暴跌）
- 联合索引遵循：最左前缀原则，等值在前、范围在后
- 禁止：`select *`、索引滥用、低基数字段建索引（性别 / 状态）
- 分页深查：禁止 `limit 100000`，用主键回溯优化
- 唯一业务（手机号 / 账号）必建 **唯一索引**，防脏数据

---

## 五、SQL 编写权威规约（阿里 Java 开发手册・数据库篇原版）
- 禁止 `SELECT *`，按需查字段
- WHERE 条件杜绝函数操作索引字段（不走索引）
- JOIN 严禁大表关联小表，控制关联层数
- IN 列表值控制数量，超长改 EXISTS / 分批
- 禁用：`ORDER BY RAND`、全文瞎查、隐式类型转换
- 事务粒度尽量小，避免长事务（锁等待、回滚膨胀）

---

## 六、分库分表 & 高可用规范（字节 / 阿里分布式标准）
- 单表数据阈值：MySQL 单表 1000 万行 / 单库不超 200G，到达即分表
- 分片规则：按用户 ID / 订单 ID 哈希分片，杜绝跨片大查询
- 读写分离：主库写、从库读，核心数据延迟兜底校验
- 分布式事务：优先最终一致性，慎用强一致（XA 笨重），用本地消息表 / 事务消息

---

## 七、安全 & 合规权威要求（等保 2.0 + 隐私法规）
- 敏感数据：手机号 / 身份证 / 银行卡 加密存储，严禁明文
- 日志脱敏、导出权限严控
- 定期备份：全量 + 增量备份，定时演练恢复
- 账号权限：最小权限原则，杜绝 root 日常业务使用
